# Compressor short-cycle guard

A small, self-contained HeishaMon ruleset (`rules.txt`) that discourages the compressor from restarting immediately after it stops, in order to reduce short-cycling and its wear on the compressor. It's a good minimal example of the timer-armed-suppression pattern and of using a sentinel value on an existing topic instead of a dedicated `#` global to track state.

For the general rules-engine language (sigils, event blocks, built-in functions, device-side validation/safety behavior) see the `heishamon-rules` skill (`Examples/Rules/SKILL.md`). Regression scenarios for this ruleset live in `tests/` and run off-device through the host-side harness (`../harness/`, see its README) — run them all with `../run_tests.sh`, and add a scenario when you change behavior.

## What the rules do

1. **Boot (`System#Boot`)** — arms `timer=1` with a 60 s delay. All topic values are populated well within that window after a cold boot (~40 s worst case), so when the timer fires, the guarded handler (block 3) can reliably tell whether a suppression window from before the reboot is still logically "active" on the pump and needs clearing. On a clean boot nothing was suppressed, the guard fails, and **no command is sent at all** — unlike an unconditional reset, which would push a spurious command to the pump on every boot and every rules-save.

2. **Suppress on compressor stop (`on @Compressor_Freq`)** — fires on every change of `@Compressor_Freq`. The whole arming condition is one `&&` chain: the compressor has just stopped (`@Compressor_Freq == 0`), no defrost cycle is in progress (`@Defrosting_State == 0`), the weather is mild (`@Outside_Temp >= 2`), and no heat request offset is already in play (`@Z1_Heat_Request_Temp == 0`, which also doubles as the re-entry guard). If all hold, it drops `@SetZ1HeatRequestTemperature` to `-5` and arms `timer=1` for `2700` s (45 min). Lowering the requested zone-1 temperature offset makes the heat curve less eager to call for heat again, keeping the compressor off longer than it otherwise would be.

3. **End of suppression (`on timer=1`)** — the single restore path, shared by all three ways a suppression can end (window expiry, boot cleanup, cold-weather override). It restores `@SetZ1HeatRequestTemperature` to `0` only if the pump actually reports the sentinel (`@Z1_Heat_Request_Temp == -5`); otherwise it stays silent, so the pump is never sent a command it doesn't need.

4. **Cold-weather override (`on @Outside_Temp`)** — if the outside temperature drops below `2 °C` while a suppression window is active (`@Z1_Heat_Request_Temp == -5`), re-arm the pending `timer=1` to fire one second from now (`setTimer(1, 1)`). Block 3 then performs the restore. Cold weather takes priority over cycle-reduction — the unit should not be kept from calling for heat once conditions turn cold. Note the override does **not** write `@SetZ1HeatRequestTemperature` itself: routing the restore through the timer handler keeps a single restore path and avoids sending the same command twice (once directly, once when the re-armed timer fires a moment later).

## Design notes for extending this pattern

- **`@Z1_Heat_Request_Temp == 0` / `== -5` as an implicit state flag.** Rather than adding a `#` persistent global just to track "is suppression active", this script reuses the read-back of the topic it already writes. That keeps the ruleset smaller (relevant given the ~10 KB best-practice limit) but only works because `-5` is otherwise not a value normal curve logic would produce for this topic — if you reuse this pattern elsewhere, pick a sentinel value that can't collide with a legitimate one.
- **One restore path.** Every way a suppression ends goes through `on timer=1`. When a state machine has exactly one place that undoes its action, you can't get duplicate or contradictory commands out of overlapping triggers — and the heat pump's slow serial link punishes redundant commands (see the skill's throughput note).
- **Timer ID `1`** is the only timer this script uses. If you merge this into a larger ruleset (e.g. alongside `Jeisha-DHW-Radiators-Rowbuffer`), pick an unused timer ID — don't reuse `1` for something else.
- **`setTimer(id, 0)` cancels — it never fires.** In the firmware's timer queue (`src/common/timerqueue.cpp`), re-arming a pending timer with `0` seconds removes it, and arming a non-pending timer with `0` is ignored. To make a pending timer fire *sooner* instead, re-arm it with a short positive delay — that's what `setTimer(1, 1)` in the `@Outside_Temp` handler does: it replaces the remaining suppression delay with 1 s, so the restore handler runs almost immediately.
- **`NULL` inside `&&` chains — engine footgun.** A bare `if @X == 0 then` is false while `@X` is still `NULL`, but the same comparison *inside an `&&` chain* (`if @X == 0 && 1 == 1 then`) evaluates truthy (verified against the engine). The one-line condition in block 2 is safe regardless, because all four topics live in the same heat pump datagram: the firmware updates the whole buffer before firing any topic event (`decode.cpp`), so when `@Compressor_Freq` fires, the other three are never `NULL` — and after a cold boot all values are populated within ~40 s anyway, well before the 60 s boot timer. If you extend the condition with values that can be `NULL` at event time (optional-PCB topics, `#` globals before their boot seeding), guard them with `isset()` first or use nested `if`s.
- **Async write caveat still applies.** `@SetZ1HeatRequestTemperature = -5;` does not make `@Z1_Heat_Request_Temp` read back as `-5` within the same firing — the guard checks in later, independent event firings (`on timer=1`, `on @Outside_Temp`), not immediately after the write, so this script does not fall into that trap.
